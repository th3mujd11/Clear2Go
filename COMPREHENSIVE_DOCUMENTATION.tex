\chapter{Executive Summary}

Clear2Go is innovative aviation communication platform resolving persistent radio communication problems between pilots and air traffic controllers. System manages real-time authorizations for aircraft operations at civilian aerodromes using mobile application and web interface.

\textbf{Core Objective:} Digitize aircraft authorization workflow (engine start, taxi, line-up, takeoff, landing) with real-time position tracking and request approval system.

\chapter{Use Case \& Architecture}

\section{System Overview}

\subsection{User Roles}
\begin{itemize}
    \item \textbf{Pilot}: Requests authorization sequentially for flight phases
    \item \textbf{Air Traffic Controller (ATC)}: Approves/denies requests and monitors live aircraft positions
\end{itemize}

\subsection{Core Workflow}

Pilot-initiated authorization request flows:
\[
\text{Motor Start} \rightarrow \text{Taxi} \rightarrow \text{Line-up} \rightarrow \text{Takeoff} \rightarrow \text{Landing}
\]

Each step blocks subsequent action until approval received. Button states:
\begin{itemize}
    \item \textcolor{red}{Red}: Initial/Denied
    \item \textcolor{orange}{Orange}: Pending (request sent)
    \item \textcolor{green}{Green}: Approved
\end{itemize}

\subsection{System Components}

\begin{enumerate}
    \item \textbf{Mobile Applications} (Android, Java)
    \begin{itemize}
        \item Pilot app (FlyActivity.java)
        \item Controller app (ControlActivity.java)
    \end{itemize}

    \item \textbf{Backend Infrastructure} (Firebase)
    \begin{itemize}
        \item Realtime Database (RTD): Authorization state, GPS coordinates
        \item Firestore: User metadata, role management
        \item Authentication: OAuth Google Sign-In
    \end{itemize}

    \item \textbf{Web Interface} (HTML/JavaScript)
    \begin{itemize}
        \item Real-time aircraft position visualization
        \item Pilot name display on right-click
    \end{itemize}

    \item \textbf{Location Services}
    \begin{itemize}
        \item GPS (device-native)
        \item Google Maps API (visualization)
    \end{itemize}
\end{enumerate}

\section{Vulnerability Assessment}

\subsection{Known Risks}

\begin{enumerate}
    \item \textbf{Hardcoded Database Path}
    \begin{itemize}
        \item Path: \texttt{"Utilizare/Aviatie/Aerodromuri/AR_AT Bucuresti/..."}
        \item Risk: Path traversal if user-supplied
        \item Mitigation: Paths hardcoded, not user-input dependent
    \end{itemize}

    \item \textbf{GPS Spoofing}
    \begin{itemize}
        \item Risk: Pilot can fake location via GPS spoofing apps
        \item Current Mitigation: None in code
        \item Recommended: Implement signature verification, multi-source verification
    \end{itemize}

    \item \textbf{Missing Input Validation}
    \begin{itemize}
        \item Request values passed directly to Firebase
        \item No sanitization against special characters
        \item Risk: If request types become user-editable, injection possible
    \end{itemize}

    \item \textbf{Firebase Rules Not Visible}
    \begin{itemize}
        \item Actual security rules not in repository
        \item Can't verify least-privilege enforcement
        \item Risk: Over-permissive rules exposing data
    \end{itemize}
\end{enumerate}

\subsection{Mitigation Strategies}

\begin{itemize}
    \item Enable Firebase Advanced Security Rules
    \item Implement server-side authorization verification
    \item Add GPS accuracy threshold validation
    \item Sanitize all user input before database operations
    \item Implement rate limiting on request submissions
    \item Log all authorization decisions for audit trail
    \item Use Custom Claims in Firebase for granular role assignment
\end{itemize}

\chapter{Scalability Analysis}

\section{Current Architecture Limitations}

\subsection{Single Aerodrome Hardcoding}

\begin{lstlisting}
// ControlActivity.java:87
positions = mData.getDatabase().getReference()
    .child("Utilizare/Aviatie/Aerodromuri/AR_AT Bucuresti/Flota/Avioane");
\end{lstlisting}

Limitation: Path hardcoded, system currently handles only Bucharest aerodrome (AR\_AT Bucuresti).

Scalability Impact: $O(1)$ per aerodrome, but requires code recompilation for new aerodromes.

\subsection{Listener Management}

\begin{lstlisting}
// FlyActivity.java: Nested ValueEventListeners
FirebaseDatabase.getInstance().getReference()
    .child("...Pornire motor")
    .addValueEventListener(new ValueEventListener() {
        // Contains nested listeners (5 levels deep)
    });
\end{lstlisting}

Problem: Each authorization creates new listener (5 nested listeners in FlyActivity)

Impact: For $n$ aircraft and $m$ authorization types, listeners $\sim O(n \times m)$

Memory cost:
\[
\text{Memory} = n \times m \times L_{base}
\]
where $L_{base}$ = base listener memory.

\subsection{Real-time Position Updates}

\begin{lstlisting}
// ControlActivity.java:161-176
Thread objUpdateThread = new Thread(new Runnable() {
    // Updates every 500ms
    updateObjectPositions1();
    Thread.sleep(Math.max(0, 500 - elapsedTime));
});
\end{lstlisting}

Update frequency: 2 Hz (every 500ms)

Database operations per aircraft per second: 2 RW ops

For $n$ aircraft:
\[
\text{Throughput} = n \times 2 \text{ ops/sec}
\]

Estimated capacity: Firebase Realtime supports ~25,000 concurrent connections. At 2 ops/sec per connection:
\[
\text{Max Aircraft} \approx 12,500
\]

\subsection{Listener Optimization}

Current: 5 nested listeners (exponential memory growth)

Optimized approach: Single listener with state machine

\begin{lstlisting}
// Consolidated listener
mDatabase.child("Avioane/" + aircraft).addValueEventListener(
    new ValueEventListener() {
        public void onDataChange(DataSnapshot snapshot) {
            AuthState state = parseAuthState(snapshot);
            updateUIBasedOnState(state);
        }
    }
);
\end{lstlisting}

Benefits:
- Single listener: $O(1)$ listeners per aircraft vs $O(m)$
- Memory reduction: 5x improvement
- Reduced Firebase bandwidth

\subsection{Database Query Indexing}

Current position queries unindexed. Add indexes:

\begin{lstlisting}
"Utilizare": {
  "Aviatie": {
    "Aerodromuri": {
      ".indexOn": ["lat", "lng", "heading"]
    }
  }
}
\end{lstlisting}

Impact: Geographic queries become $O(\log n)$ instead of $O(n)$

\subsection{Caching Strategy}

Implement local caching for frequently accessed data:

Map<String, LatLng> positionCache = new HashMap<>();

\chapter{Edge Cases \& Error Handling}

\section{Network Failures}

\subsection{Offline Operation}

Firebase enables offline persistence by default. Current implementation:

\begin{lstlisting}
// Automatic: Firebase maintains offline queue
mDatabase.child("aircraft").setValue(data);
// Data queued locally, sent when online
\end{lstlisting}

\textbf{Edge Case 1}: Pilot requests authorization while offline

\begin{itemize}
    \item Behavior: Request queued locally
    \item Resolution: Sent when connection restored
    \item Risk: Stale authorization state (controller approved while offline)
    \item Mitigation: Display "queued" state to user, timestamp validation
\end{itemize}

\textbf{Edge Case 2}: Controller app offline during request approval

\begin{itemize}
    \item Behavior: Cannot approve/deny
    \item Impact: Pilots stuck in "pending" state
    \item Mitigation: Automatic timeout (recommend 5min) resets request to red
\end{itemize}

\subsection{Network Latency}

High-latency scenarios (3G, rural areas):

\begin{itemize}
    \item GPS updates delayed 2-5 seconds
    \item Controller sees stale aircraft positions
    \item Mitigation: Extrapolate position using heading + speed, show "updated X seconds ago"
\end{itemize}

\section{GPS Errors}

\subsection{GPS Accuracy Issues}

\begin{lstlisting}
// FlyActivity.java: No accuracy validation
binding.alt.setText(String.valueOf(
    (int)((int)location.getAltitude()*3.28084)) + "ft"
);
\end{lstlisting}

\textbf{Edge Case}: GPS altitude wildly inaccurate (±200m error possible)

\begin{itemize}
    \item GPS altitude accuracy: 5-20m typical, up to 100m in urban/indoor
    \item Barometric pressure more reliable for aviation
    \item Current code: No validation
    \item Risk: Display of incorrect altitude to controller
    \item Mitigation: Use barometer sensor + GPS fusion, set confidence threshold
\end{itemize}

\subsection{GPS Loss of Signal}

\begin{lstlisting}
// No timeout if GPS stops updating
locationManager.requestLocationUpdates(
    LocationManager.GPS_PROVIDER, 0, 0, this);
\end{lstlisting}

\textbf{Edge Case}: Tunnel/urban canyon blocks GPS

\begin{itemize}
    \item Behavior: Last known position remains on controller map
    \item Risk: Outdated position for 1-5+ minutes
    \item Mitigation: Timestamp position updates, flag stale data (\textgreater 30s) with warning
\end{itemize}

\section{Authorization State Conflicts}

\textbf{Edge Case}: Pilot presses button twice rapidly

\begin{lstlisting}
binding.engOn.setOnClickListener(...);
// No debouncing - click handler fires twice
rq.child(avion).child("Pornire motor").setValue(false); // x2
\end{lstlisting}

\begin{itemize}
    \item Behavior: Duplicate request sent
    \item Firebase: Idempotent, deduplicates same value
    \item Risk: Minor (no duplicate in DB)
    \item Mitigation: Add clickListener debounce, disable button during request
\end{itemize}

\textbf{Edge Case}: Simultaneous approvals from multiple controllers

\begin{itemize}
    \item Firebase enforces last-write-wins
    \item If Controller A approves (true), Controller B denies (false)
    \item Final state: false (deny)
    \item Risk: Safety (deny overrides approve)
    \item Mitigation: Lock-based authorization (only one controller can approve per step)
\end{itemize}

\section{Data Consistency Issues}

\textbf{Edge Case}: GPS coordinates malformed in database

\begin{lstlisting}
// No null check
double lat = planeSnapshot.child("lat").getValue(Double.class);
LatLng location = new LatLng(lat, lng); // NPE if null
\end{lstlisting}

\begin{itemize}
    \item Behavior: NullPointerException in ControlActivity
    \item Risk: Controller app crash
    \item Mitigation: Add null checks, default to last-known-good position
\end{itemize}

\begin{lstlisting}
// Improved:
Double lat = planeSnapshot.child("lat").getValue(Double.class);
if (lat == null || lng == null) {
    lat = lastKnownLat;
    lng = lastKnownLng;
}
\end{lstlisting}

\section{Memory \& Resource Management}

\textbf{Edge Case}: Rapid activity switching (Pilot rotates phone, navigates back/forth)

\begin{lstlisting}
// FlyActivity exit doesn't remove listeners
binding.exit.setOnClickListener(...);
locationManager.removeUpdates(FlyActivity.this);
// But ValueEventListeners NOT removed
\end{lstlisting}

\begin{itemize}
    \item Behavior: Listeners continue firing in background
    \item Risk: Memory leak (5 nested listeners × 100 app switches = unbounded)
    \item Mitigation: Call \texttt{removeEventListener()} in onDestroy()
\end{itemize}

\begin{lstlisting}
@Override
protected void onDestroy() {
    super.onDestroy();
    if (avionRef != null) {
        avionRef.removeEventListener(valueEventListener);
    }
}
\end{lstlisting}

\section{Network Efficiency}

\subsection{Firebase RTD Operations}

\begin{itemize}
    \item \textbf{Reads}: Location updates (5 bytes lat/lng) × 2 Hz × 500 aircraft = 5 KB/sec
    \item \textbf{Writes}: Aircraft position (15 bytes) × 2 Hz = 30 bytes/sec per aircraft
    \item Total bandwidth: ~200 KB/sec (500 aircraft)
\end{itemize}

\textbf{Optimization}: Use batch writes, delta compression

\begin{lstlisting}
// Current: Write full position per update
avionData.child("lat").setValue(latitude);
avionData.child("lng").setValue(longitude);
avionData.child("heading").setValue(heading);

// Optimized: Single batch write
Map<String, Object> updates = new HashMap<>();
updates.put("lat", latitude);
updates.put("lng", longitude);
updates.put("heading", heading);
avionData.updateChildren(updates); // Single operation
\end{lstlisting}

\subsection{Database Costs}

Firebase RTD pricing (2024):
- \$1 per GB downloaded
- \$5 per GB uploaded

For 500 aircraft, 8 hours:
\[
\text{Cost} = (500 \text{ aircraft} \times 8 \text{ hours} \times 3600 \text{ sec/hour} \times 2 \text{ ops/sec} \times 15 \text{ bytes}) / (10^9) \times \$1
\]
\[
= 432 \text{ GB/day} = \$432/day
\]

Optimization savings (batch writes, 5x reduction):
\[
\text{Optimized Cost} = \$86.40/day
\]
